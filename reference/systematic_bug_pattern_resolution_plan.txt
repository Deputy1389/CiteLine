# CiteLine: Systematic Bug Pattern Resolution Plan

## Executive Summary

Six recurring anti-patterns are causing systemic issues in the CiteLine pipeline:

1. **Pipeline fragmentation** (ROOT CAUSE) - Multiple entry points with different quality gates
2. **Config doesn't flow** - Settings are defined but not consistently applied
3. **Outputs not wired** - Data is computed but never reaches the frontend
4. **Type mismatches** - SQLAlchemy JSON columns cause silent runtime failures
5. **Bad text propagates** - Quality gates run too late to prevent downstream corruption
6. **Attorney readiness not gated** - Production exports without quality checks

---

## Pattern 1 (ROOT CAUSE): Pipeline Fragmentation

### The Problem

There are THREE separate pipeline entry points with DIFFERENT quality outcomes:

| Entry Point | Called By | Quality Gates | Used For |
|-------------|-----------|---------------|----------|
| `pipeline.py:run_pipeline()` | `runner.py`, `runner_once.py`, `bulk_ingest_*.py` | **NONE** | Production |
| `run_case.py:run_case()` | CLI, `run_critic_loop.py`, `eval_invariant_robustness.py` | ALL (litigation, LUQA, attorney) | Evaluation |
| `run_sample_pipeline()` (inline) | `run_case.py` internally | Relies on wrapper | Simplified eval |

### The Architecture Gap

```
┌─────────────────────────────────────────────────────────────────┐
│                   PRODUCTION PATH ❌                              │
│  runner.py → pipeline.py:run_pipeline() → persist               │
│  • NO quality gates                                             │
│  • NO attorney readiness check                                  │
│  • Fail cover PDF never written                                 │
│  • Bad outputs reach lawyers                                    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    EVAL PATH ✅                                   │
│  run_case.py:run_case()                                         │
│      ↓                                                          │
│  run_sample_pipeline() (DUPLICATE code, not pipeline.py!)        │
│      ↓                                                          │
│  build_litigation_checklist()  ← GATE EXISTS                    │
│  build_luqa_report()            ← GATE EXISTS                   │
│  build_attorney_readiness_report() ← GATE EXISTS               │
│      ↓                                                          │
│  _write_fail_cover_pdf()        ← GATE EXISTS                  │
└─────────────────────────────────────────────────────────────────┘
```

### Consequences

1. **Production can export outputs that would fail the eval gates** - Lawyer gets bad output
2. **Fixes land in one path and silently don't apply to the other** - Maintenance nightmare
3. **"Lawyer-ready" is not enforceable** - No single source of truth for gating

### Root Causes

| Location | Issue |
|----------|-------|
| `scripts/eval_sample_172.py:165-350` | Inline pipeline (not calling `pipeline.py`) |
| `scripts/run_case.py:195` | Calls `run_sample_pipeline()`, not `pipeline.py` |
| `apps/worker/pipeline.py` | No quality gates after rendering |
| No single source | Three entry points, not one |

### Non-Negotiable Invariant

> "Any exportable artifact must pass the same gates regardless of entry point. Same input, same render, same gating, same fail behavior."

### Recommended Fix: Consolidate to One Pipeline

#### Implementation Shape (cleanest)

1. **Extract a shared core**: Create `pipeline_core.py` (or refactor `pipeline.py`) with:
   - `run_pipeline_core(input, config) -> PipelineResult`
   - Produces the same intermediate objects used by both production and eval

2. **Centralize gating**: Create `quality_gates.py` with:
   - `run_quality_gates(result, gate_profile) -> GateReport`
   - `gate_profile` covers: litigation, LUQA, attorney readiness

3. **Single export decision**: In all entry points, before writing final artifacts:
   ```python
   if gates fail:
       write fail cover PDF + persist gate report + mark run non-exportable
   else:
       export
   ```

#### Migration Phases

| Phase | Action |
|-------|--------|
| **Phase 1 (now)** | Wrapper layer (Option C) - production calls `run_pipeline_with_gates()` to stop bleeding |
| **Phase 2** | Delete `run_sample_pipeline()` and have `run_case.py` invoke production pipeline |
| **Phase 3** | Delete wrapper when consolidation complete |

#### Definition of Done

- [ ] One pipeline core exists
- [ ] One gate runner exists  
- [ ] `run_case.py` uses production pipeline core
- [ ] `quality_gates.py` is the only gate implementation
- [ ] Production runner uses same gate profile as eval
- [ ] CI test fails if any entry point bypasses gates

## Pattern 2: Config Doesn't Flow

### Symptom
- User changes PT mode in UI → output unchanged
- User sets confidence threshold → silently overwritten
- LLM model override → worker uses different model

### Root Causes

| Location | Issue |
|----------|-------|
| `apps/api/routes/runs.py:27` | `CreateRunRequest` defaults `pt_mode="aggregate"` and `event_confidence_min_export=40` |
| `packages/shared/models/domain.py:30,33` | `RunConfig` defaults `pt_mode="per_visit"` and `event_confidence_min_export=30` |
| Git history (main:195) | Prior commit "Force high-density pt_mode=per_visit" suggests hardcoded override existed |
| `apps/api/routes/runs.py:25` | Request model has hardcoded defaults different from domain model |

### Fix Plan

1. **Unify default values**
   - Create shared constants or import `RunConfig` defaults into `CreateRunRequest`
   - Ensure API defaults match Worker defaults exactly

2. **Add config validation**
   - In `pipeline.py:143`, after loading config from DB, add validation that all expected keys exist
   - Log warning if config is missing expected fields

3. **Add config passthrough verification**
   - Create a test that sets a unique config value and verifies it appears in the output
   - Automate this in CI for all RunConfig fields

## Pattern 4: SQLAlchemy Column Type Mismatches

### Symptom
- Mysterious 500 errors with no useful message
- Page worked yesterday, breaks after run completes
- API crashes on specific matter but works on others (completed run with metrics)

### Root Causes

| Location | Issue |
|----------|-------|
| `pipeline_persistence.py:45` | Stores `run_record.metrics.model_dump_json()` (a string) into `JSON` column |
| `pipeline.py:142` | Reads `config_json` with `json.loads()` expecting string |
| `runs.py:110,146,186` | Reads `metrics_json` directly expecting dict |

### The Bug

```python
# Writing side (pipeline_persistence.py:45):
run_row.metrics_json = run_record.metrics.model_dump_json()  # Returns STRING

# Reading side (runs.py:110):
metrics = r.metrics_json  # Expects DICT

# Reading side (pipeline.py:142):  
config_dict = json.loads(run_row.config_json)  # Expects STRING
```

SQLAlchemy's JSON column accepts both strings and dicts, but the code inconsistently expects one or the other.

### Fix Plan

1. **Standardize on dict for JSON columns**
   - Change `model_dump_json()` → `model_dump()` in persistence layer
   - Or standardize on string and always use `json.loads()` on read

2. **Add type assertions**
   - After reading JSON columns, assert type is dict
   - Provide clear error message if type mismatch detected

3. **Migration script**
   - Clean existing data: convert string-stored JSON to actual JSON objects
   - Run on production DB

4. **Add runtime type checks**
   - Create helper function `get_json_column(row, attr_name)` that validates and returns dict
   - Use everywhere JSON columns are read

## Pattern 5: Text Quality Verification Happens Too Late

### Symptom
- `(cid:###)` strings in chronology
- Every provider showing as "Unknown"
- Events extracted from wrong document sections
- Bad OCR gets cached and reused

### Root Cause

Current flow:
```
1. detect_providers()         ← Runs on ALL text, including garbage
2. extract_clinical_events() ← Events extracted from garbage
3. extract_imaging_events()  ← More garbage events
4. extract_pt_events()      ← ...
5. [QUALITY GATE]            ← is_garbage() check - TOO LATE!
6. filter_for_export()       ← Filters garbage, but providers already created
```

Quality gate filters facts but:
- Already created from garbage Providers were text
- "Unknown Provider" sentinel created
- Downstream steps have no signal to know text was bad

### Fix Plan

1. **Move quality gate earlier**
   - After text acquisition (step 02), run quality check on each page
   - Flag low-quality pages
   - Pass quality flag to provider detection

2. **Provider detection with quality awareness**
   - If page text is garbage, don't extract providers from it
   - Or weight provider confidence by text quality score

3. **Quality-aware event extraction**
   - Skip event extraction from low-quality pages
   - Or mark events from low-quality pages with low confidence

4. **OCR cache invalidation**
   - Add quality score to OCR cache key
   - Re-run OCR if quality threshold not met

5. **Add quality metrics to output**
   - Include page quality stats in evidence_graph
   - Show in audit view which pages had quality issues

## Pattern 6: Attorney Readiness Not Gated in Production

### The "Lawyer-Ready" Definition

A run is "lawyer-ready" if it meets these requirements:

1. **Source-verifiable in under 5 seconds** - Every claim must have a clear citation
2. **Every sentence survives cross-examination** - No weak or uncited facts
3. **No embarrassment in deposition** - Defense can't poke holes in it

### Current State: Evaluation Has the Gate, Production Doesn't

The codebase already has a sophisticated attorney readiness check in `apps/worker/lib/attorney_readiness.py`:

| Check | Threshold | Description |
|-------|-----------|-------------|
| `AR_UNCITED_FACT_ROWS` | ≤5% | Every fact must have a citation |
| `AR_FACT_DENSITY_LOW` | ≥60% dense | Facts must be substantive |
| `AR_REQUIRED_BUCKETS_MISSING` | Must have ED/MRI/ORTHO/PROCEDURE | Key milestone types present |
| `AR_MISSING_REQUIRED_SECTIONS` | Required sections | Structured output |

#### What's Working (in `scripts/run_case.py` - evaluation pipeline):

- Line 131: Returns early if all checks pass
- Line 156-161: Writes "ATTORNEY READINESS FAIL" cover page when checks fail
- Line 270-272: Tracks `attorney_ready_pass` in scorecard
- `_write_fail_cover_pdf()` prepends failure page to output PDF

#### The Critical Gap (in `apps/worker/pipeline.py` - production pipeline):

- Line 271: Runs `run_litigation_review()` but **doesn't block on its result**
- Line 287-288: Only validates schema, not quality
- **No fail cover page written**
- **No attorney readiness score surfaced to user**
- **Bad outputs reach the lawyer**

### Root Cause

| Location | Issue |
|----------|-------|
| `pipeline.py:265-291` | Runs export but doesn't check quality before marking complete |
| `pipeline.py:287` | Only schema validation (`validate_output`), no quality gate |
| No location | No call to `build_attorney_readiness_report()` in production pipeline |

### Fix Plan

1. **Run attorney_readiness in production pipeline**
   - After rendering exports, call `build_attorney_readiness_report()`
   - Pass required context (report_text, page_text_by_number, projection_entries)

2. **Block exports on failure**
   - If `attorney_ready_pass == False`, write fail cover page to PDF
   - Set run status to "failed" or "needs_review" instead of "success"
   - Prevent lawyer from downloading until score passes

3. **Surface score in UI**
   - Add `attorney_ready_score` to RunResponse API
   - Display prominently: "Score: 65/100 - DO NOT USE"
   - Show specific failure reasons

4. **Tighten thresholds for deposition safety**
   - Change uncited threshold from 5% to 0%
   - Increase fact density from 60% to 75%
   - Add explicit "deposition_ready" flag

5. **Add source verification links**
   - In PDF, make citations clickable to jump to source page
   - In UI, show "Verify Source" button that opens PDF at page

## Prevention Measures

### Config Contract Test

```python
def test_config_passthrough():
    """Every RunConfig field should appear in output."""
    config = RunConfig(...)
    result = run_pipeline(config)
    for field in RunConfig.model_fields:
        assert field in result.run_config_used
```

### Extension Registry

```python
# EvidenceGraphExtensions.py
WRITTEN_KEYS = [
    "patient_partitions",
    "extraction_metrics", 
    "quality_gate",
    "event_weighting",
    "providers_normalized",
    "coverage_spans",
    "missing_records",
    "billing_lines",
    "specials_summary",
    "paraleger_chronology",
]

READ_KEYS = [track all reads]
```

### Type-Safe JSON Columns

```python
def get_json_attr(row, attr: str) -> dict:
    val = getattr(row, attr)
    if isinstance(val, str):
        return json.loads(val)
    if isinstance(val, dict):
        return val
    raise TypeError(f"{attr} is {type(val)}, expected dict")
```
